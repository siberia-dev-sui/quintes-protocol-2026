Quintes Protocol — System Overview

Version: 1.0  |  Date: February 10, 2026  |  Network: Arbitrum One (Mainnet)

1. What Does the System Do?

Quintes Protocol uses iExec Web3Mail to create a privacy-preserving whitelist registration. Users join the whitelist by encrypting their email on-chain and receiving a confirmation — without Quintes ever seeing their email address.

2. Registration Flow (Step by Step)

When a user clicks [Join Verified Whitelist], the following 4-step process happens:

3. Data Storage — What We Collect

3.1 Supabase Database

The whitelist_participants table stores the following:

Field Type Purpose Contains PII?
id UUID Internal row identifier No
created_at Timestamp Registration date No
wallet_address Text Ethereum address (public) No
protected_data_address Text iExec encrypted data reference No
referral_code Text Unique referral code No
referred_by Text Referrer's code (if any) No
ip_address Text Anti-bot detection Yes
user_agent Text Browser fingerprint Yes
status Text Always "active" No
⚠️ IMPORTANT: We do NOT store the user's email address. The email is encrypted by iExec and stored on-chain inside a Trusted Execution Environment (TEE). Only the Web3Mail application can access it — and only with the user's explicit permission.

3.2 On-Chain Data (iExec / Arbitrum One)

Data Where Who Controls It
Encrypted email iExec TEE (on-chain NFT) User (owner)
Access permissions iExec Smart Contract User (can revoke anytime)
RLC deposit iExec Account Contract User (can withdraw)

5. Admin Dashboard

Accessible at /admin/dashboard (requires Supabase authentication).

Features:

  • Total registered users count
  • Daily signup metrics
  • Recent signups table with wallet addresses, referral codes, and timestamps
  • Status filter (active/inactive)

Access:

  • Login at /admin/login with admin credentials
  • Only authenticated users can read data (RLS policy)
  • Anyone can INSERT (public whitelist registration)
